# 基于 Elasticsearch 的南开大学新闻通知垂直搜索引擎设计与实现

## 一、项目背景与目标

南开大学官方网站及其各二级单位网站每天发布大量新闻、通知、公告、讲座、就业和招生信息。这些信息分散在数十个独立站点中，缺乏统一的检索入口，师生获取目标信息的效率较低。

本项目旨在构建一个面向南开大学公开网页资源的垂直搜索引擎，实现以下目标：

1. **统一检索**：将分散在各站点的新闻、通知、公告、讲座、就业、招生信息汇聚到单一搜索入口，支持全文检索。
2. **高效查询**：借助 Elasticsearch 的倒排索引和 IK Analyzer 中文分词，提供毫秒级搜索响应。
3. **多样化搜索**：支持站内查询、文档查询、短语查询、通配查询等多种搜索模式。
4. **网页快照**：保存抓取时的网页原始内容，提供快照查看功能，避免源链接失效导致信息丢失。
5. **个性化服务**：基于用户身份、学院归属和兴趣标签，对搜索结果进行个性化排序修正，并推荐相关内容。
6. **查询日志分析**：记录用户查询行为，为检索效果评估和个性化推荐提供数据支撑。

| 职责 | MySQL | Elasticsearch |

| 数据一致性 | 写入后立即一致 | 近实时（refresh interval ~1s） |

系统分为五层架构：

- **数据采集层**：Scrapy 爬虫负责网页抓取，解析模块负责 HTML 正文提取和附件文档内容解析，产出结构化数据写入 MySQL。
- **数据存储层**：MySQL 作为权威数据源存储所有结构化数据；Elasticsearch 存储文档索引副本，专责全文检索。
- **数据访问层**：SQLAlchemy ORM 封装 MySQL 访问；Elasticsearch Python Client 封装 ES 查询。
- **业务服务层**：SearchService 封装搜索逻辑，RecommendService 封装推荐逻辑，LogService 封装日志记录。路由层不直接操作 ES 客户端。
- **API 网关层**：FastAPI 提供 RESTful 接口，按模块拆分路由（search、documents、snapshots、recommend）。
- **前端展示层**：React + TypeScript + Ant Design 构建的用户界面。

## 六、排序模型设计

### 6.1 排序公式

系统采用**两阶段排序 + 加权融合**策略：

```
final = 0.60 × BM25 + 0.20 × VSM + 0.15 × PageRank + 0.05 × PERS
```

| 权重 | 得分 | 来源 | 说明 |
|------|------|------|------|
| 60% | BM25 | Elasticsearch 原生评分 | 基于词频和文档长度 |
| 20% | VSM | TF-IDF cosine similarity | jieba 分词后对 title/content/anchor_text 计算 |
| 15% | PageRank | 链接分析 | 预留字段，当前为 0 |
| 5% | PERS | 用户画像 | 角色关键词 / 学院 / 兴趣加权 |

### 6.2 VSM 实现

`services/vsm_reranker.py`：
- 接收 ES 召回 top-k 文档（不修改 ES 查询）
- jieba 分词 → 去停用词 → TF-IDF 向量
- 对 title(0.5) + content(0.4) + anchor_text(0.1) 三字段加权 cosine similarity
- min-max 归一化到 [0, 1]

### 6.3 个性化服务

`services/personalization_service.py`：
- 5 种角色各有独立关键词库（undergraduate/graduate/teacher/job_seeker/visitor）
- 角色关键词命中 title(+2.0) / content(+0.5) 加分
- 学院名称命中 title(+2.0) / content(+0.5) 加分
- 兴趣标签命中 title(+2.0) / content(+0.5) / category(+1.5) 加分
- 未登录用户 preference_score = 0

## 七、爬虫模块设计

### 7.1 框架选型

基于 Scrapy 2.11 构建爬虫模块。Scrapy 提供异步引擎、自动去重、请求调度、中间件管道等基础设施，开发者只需编写 Spider 解析逻辑和 Pipeline 持久化逻辑。

### 6.2 Spider 设计

采用**配置驱动双爬虫**架构，通过 `config/sites.py` 统一管理 40 个 NKU 子域名：

| Spider | 文件 | 策略 | 适用场景 |
|--------|------|------|----------|
| `news` | `news_spider.py` | 正则模式匹配 | 已知 URL 结构的站点（如 news.nankai.edu.cn），解析列表页 → 逐个详情页 |
| `broad` | `broad_spider.py` | BFS 宽泛抓取 | 任意 NKU 子域名，跟踪所有内部链接，提取每个页面内容 |

站点配置结构 (`config/sites.py`)：
```python
SITES = [_site("chem", "chem.nankai.edu.cn", "chemistry", max_pages=8000), ...]
# 40 站点, Tier1(8K) × 6 + Tier2(5K) × 21 + Tier3(3K) × 13 = 211K 容量
```

命令行参数：
- `-a max_pages=N`：每站最大页数
- `-a global_max=N`：全局上限
- `-s DEPTH_LIMIT=N`：链接深度（默认 10）

### 6.3 礼貌抓取策略

遵循 robots.txt 协议，在 `settings.py` 中配置：

```python
ROBOTSTXT_OBEY = True
DOWNLOAD_DELAY = 2            # 请求间隔 2 秒
CONCURRENT_REQUESTS = 8       # 单站点并发数
CONCURRENT_REQUESTS_PER_DOMAIN = 4
USER_AGENT = "NankaiSearchBot/1.0 (+https://github.com/nankai-search)"
```

### 6.4 增量抓取

通过数据库记录已抓取 URL，新抓取前查询 URL 是否已存在、内容是否有更新，避免重复抓取未变化的页面。

## 八、网页解析模块设计

### 7.1 解析流程

```
原始 HTML
    │
    ▼
BeautifulSoup4 + lxml（HTML 解析为 DOM 树）
    │
    ├──→ 提取标题：<title> 或 <h1> 标签
    ├──→ 提取发布时间：正则匹配日期格式
    ├──→ 提取来源：meta 标签或页面标识
    └──→ 正文提取：readability-lxml 算法
             │
             ▼
         清洗后的纯文本正文
```

### 7.2 正文提取

使用 readability-lxml 库提取网页正文。该库实现了 Readability 算法，能够自动识别网页中的正文区域，去除导航栏、侧边栏、页脚等噪声内容。

对于 readability 无法正确处理的页面（如表格布局的旧页面），回退到基于 BeautifulSoup4 的自定义规则提取。

### 7.3 数据清洗

- 移除 HTML 标签和脚本
- 合并多余空白字符
- 统一字符编码为 UTF-8
- 过滤少于 50 字符的无效正文

## 十、Elasticsearch 索引设计

### 10.1 索引名称

`nankai_documents`

### 10.2 索引映射（Mapping）

```json
{
  "settings": {
    "number_of_shards": 1,
    "number_of_replicas": 1,
    "analysis": {
      "analyzer": {
        "ik_max_word_analyzer": {
          "type": "custom",
          "tokenizer": "ik_max_word"
        },
        "ik_smart_analyzer": {
          "type": "custom",
          "tokenizer": "ik_smart"
        }
      }
    }
  },
  "mappings": {
    "properties": {
      "id":              { "type": "long" },
      "url":             { "type": "keyword" },
      "title":           { "type": "text", "analyzer": "ik_max_word_analyzer", "search_analyzer": "ik_smart_analyzer" },
      "content":         { "type": "text", "analyzer": "ik_max_word_analyzer", "search_analyzer": "ik_smart_analyzer" },
      "site_type":       { "type": "keyword" },
      "source_domain":   { "type": "keyword" },
      "department":      { "type": "keyword" },
      "publish_time":    { "type": "date", "format": "yyyy-MM-dd HH:mm:ss" },
      "has_attachment":  { "type": "boolean" },
      "attachment_content": { "type": "text", "analyzer": "ik_max_word_analyzer", "search_analyzer": "ik_smart_analyzer" },
      "suggest":         { "type": "completion" }
    }
  }
}
```

### 10.3 分词策略

| 阶段 | 分词器 | 说明 |
|------|--------|------|
| 索引时 | `ik_max_word` | 尽可能多地切分词语，提高召回率。例如"南开大学" → "南开大学"、"南开"、"大学" |
| 搜索时 | `ik_smart` | 粗粒度切分，提高精度。例如"南开大学" → "南开大学" |
| 自动补全 | `completion` | suggest 字段用于搜索框输入提示 |

### 10.4 索引同步策略

- **全量索引**：首次部署时，将 MySQL pages 表中全部数据批量写入 ES。
- **增量索引**：定时任务（每 5 分钟）查询 MySQL 中 `updated_at > last_sync_time` 的记录，同步至 ES。
- **删除同步**：MySQL 中标记为删除的记录，在 ES 中执行 `delete` 操作。

## 十一、查询服务设计

查询服务由 `SearchService` 类统一封装，位于 `backend/app/services/search.py`。所有查询业务逻辑收敛于此层，路由层仅负责参数校验和响应序列化。

### 11.1 站内查询

用户输入关键词，在 title 和 content 字段中执行全文检索。

```python
# 核心查询 DSL
{
  "query": {
    "bool": {
      "must": [
        { "multi_match": { "query": keyword, "fields": ["title^3", "content"] } }
      ],
      "filter": [
        { "term": { "site_type": site_type } }  # 可选：按站点类型筛选
      ]
    }
  },
  "highlight": {
    "fields": {
      "title": { "number_of_fragments": 0 },
      "content": { "fragment_size": 150, "number_of_fragments": 3 }
    }
  },
  "from": offset,
  "size": page_size
}
```

- title 字段权重设为 3，content 字段权重为 1。
- 筛选条件（site_type、source_domain、时间范围）放入 `filter` 子句，不参与评分。
- 高亮返回匹配片段，前端用于展示摘要。

### 11.2 文档查询

针对附件文档内容（attachment_content 字段）的检索。查询时联合搜索 content 和 attachment_content 字段：

```python
"fields": ["title^3", "content", "attachment_content"]
```

### 11.3 短语查询

支持用户使用双引号进行精确短语匹配，如 `"南开大学计算机学院"`。

```python
{
  "query": {
    "match_phrase": {
      "content": {
        "query": phrase,
        "slop": 0       # 词项间距，0 表示严格连续
      }
    }
  }
}
```

同时支持混合查询：无引号部分仍执行普通 multi_match，引号部分执行 match_phrase，两者以 bool should 组合。

### 11.4 通配查询

支持 `*` 和 `?` 通配符进行模糊匹配。

```python
{
  "query": {
    "wildcard": {
      "title": { "value": pattern }
    }
  }
}
```

为防止性能问题，通配查询限制：
- 不允许通配符在字符串首位
- 返回结果上限 500 条

### 11.5 查询日志

每次搜索请求完成后，`LogService` 异步记录查询日志至 MySQL `query_logs` 表：

```python
# LogService.record_query()
log_entry = QueryLog(
    user_id=current_user_id,
    query_text=query,
    result_count=total_hits,
    search_type=search_type,
    ip_address=request.client.host
)
db.add(log_entry)
await db.commit()
```

日志用于：
1. 热门查询词统计（Top N 热搜词）
2. 零结果查询分析（发现爬虫覆盖盲区）
3. 用户搜索兴趣画像构建

### 11.6 网页快照

爬虫抓取时将原始 HTML 存入 `pages.raw_html` 字段。快照查看流程：

1. 用户点击"快照"链接，前端请求 `GET /api/documents/{id}/snapshot`
2. 后端从 MySQL `raw_html` 字段读取原始 HTML
3. 对 HTML 做安全清洗（移除 script 标签、外部资源引用）
4. 返回清洗后的 HTML，前端在 iframe 或独立页面中渲染

快照页面顶部显示提示条："这是 2024-01-15 抓取的网页快照，原页面可能已更新。"

## 十二、个性化查询设计

### 12.1 用户画像

每位已登录用户拥有一个用户画像（User Profile），由以下维度构成：

| 维度 | 来源 | 说明 |
|------|------|------|
| 学院归属 | users.department | 如"计算机学院"，该学院相关新闻权重提升 |
| 兴趣标签 | users.interest_tags | JSON 数组，如 ["人工智能", "数据库", "信息检索"] |
| 搜索历史 | search_history 表 | 近期搜索关键词，反映当前关注焦点 |
| 点击行为 | 隐式反馈 | 因初期简化，暂不采集；后期可加入 |

### 12.2 排序修正

在 ES 原生 BM25 评分基础上，引入个性化权重修正因子。最终排序分数为：

```
final_score = BM25_score × (1 + personal_boost)
```

`personal_boost` 计算规则：

1. **学院匹配**：文档内容提及用户所属学院 → +0.2
2. **兴趣标签匹配**：文档 title 或 content 命中用户兴趣标签 → 每个命中 +0.1（上限 0.3）
3. **搜索历史关联**：文档与近期搜索词相关 → +0.1
4. **时效性衰减**：发布时间超过 30 天的文档 → -0.1；超过 180 天 → -0.2

实现方式——使用 ES 的 `function_score` 查询，在 `script_score` 中计算自定义评分逻辑：

```python
{
  "query": {
    "function_score": {
      "query": { "bool": { "must": [...] } },
      "functions": [
        {
          "script_score": {
            "script": {
              "source": """
                double boost = 0.0;
                if (doc['department'].size() > 0 && params.userDept != null
                    && doc['department'].value.contains(params.userDept)) {
                  boost += 0.2;
                }
                // 时效衰减
                long daysAgo = (System.currentTimeMillis() - doc['publish_time'].value.millis)
                               / (1000 * 60 * 60 * 24);
                if (daysAgo > 180) boost -= 0.2;
                else if (daysAgo > 30) boost -= 0.1;
                return _score * (1 + boost);
              """,
              "params": { "userDept": "计算机学院" }
            }
          }
        }
      ]
    }
  }
}
```

未登录用户使用默认 BM25 排序，不施加个性化修正。

## 十三、个性化推荐设计

### 13.1 推荐策略

采用"查询日志 + 内容相似度"的简化推荐方案，分为两种推荐场景：

**场景一：搜索无结果或结果较少时**

推荐逻辑：
1. 对用户查询词做分词处理
2. 将分词结果在 query_logs 中查找相似查询
3. 取相似查询对应的高点击结果作为推荐

**场景二：搜索结果页侧边栏"相关推荐"**

推荐逻辑（基于内容相似度）：
1. 取当前搜索结果前 5 条的文档内容
2. 使用 ES 的 `more_like_this` 查询，以这些文档为输入，查找内容相似的其他文档
3. 按相似度排序返回 Top 10

```python
{
  "query": {
    "more_like_this": {
      "fields": ["title", "content"],
      "like": [{ "_id": doc_id_1 }, { "_id": doc_id_2 }],
      "min_term_freq": 1,
      "max_query_terms": 12,
      "min_doc_freq": 1
    }
  }
}
```

**用户维度推荐（已登录）**：

1. 获取用户近 30 天的搜索历史关键词
2. 为每个关键词执行 ES 查询，取 Top 3 结果
3. 汇总去重后，按出现频次排序，返回推荐列表

### 13.2 推荐服务接口

```
GET /api/recommend?type=similar&doc_id=123    # 基于文档的相似推荐
GET /api/recommend?type=personal               # 基于用户画像的个性化推荐
GET /api/recommend?type=hot                    # 热门内容推荐（基于查询日志）
```
